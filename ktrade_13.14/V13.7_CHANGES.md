# KTrade — V13.7 (N-cycle BUY confirmation)

Implements the one piece of the Kubera-comparison feedback that (a) is genuinely
worth building, (b) needs NO news feed, and (c) is fully verifiable offline. It
directly fixes the worst failure in that log: copilot bought FCEL on one scan, then
said "SKIP FCEL" on the next. With confirmation on, a one-scan blip never reaches
the broker.

## New: agent/buy_confirmation.py + run_cycle wiring (OFF by default)
`KTRADE_BUY_CONFIRM_CYCLES` (default **1** = off, no behaviour change):
- **2+** = a name must clear hard gates + strategy signal in N CONSECUTIVE scan
  cycles before it executes. Sits AFTER the hard gates (so it gates only
  rule-approved BUYs) and is logged as `held_for_confirmation` while it waits.
- A name that's approved, then NOT approved next cycle, then approved again does NOT
  confirm — the streak restarts (exactly the FCEL case).
- `KTRADE_BUY_CONFIRM_MAX_GAP_MINUTES` (default 20) breaks a streak across long
  pauses (overnight, restart) so a pass from a prior session can't count today.

## HONEST trade-off (stated plainly, why it's off by default)
Confirmation delays entry by (N-1) cycles — you give up the first-cycle move. That's
a fair trade for INTRADAY scanning (a 5m cycle = a 5-minute wait) but a poor one for
DAILY scanning (a full day late, which hurts momentum entries). So this is an
intraday-oriented control; leave it off for daily. The 20-minute gap default suits
cycles <= 15m; raise it for 30m/daily. I did not pretend it's free upside.

## On the rest of that feedback — what I did NOT build, and why
- **CatalystVerifier (the headline rec):** blocked by the constraint flagged since
  v13.0 — **KTrade has no live news feed.** I can build the scoring LOGIC (ticker/
  company/exchange match, article age, direct-vs-indirect, ETF/macro down-weighting),
  but it is INERT until a real news source (Finnhub/Polygon news API) is wired on the
  VPS — which needs keys and can't be verified here. It is NOT the "one quick change"
  the feedback implies; it's a data integration (your side) + a verifier (buildable
  later). I won't ship a news-dependent module that can't actually run.
- **#4 "LLM only after hard gates":** already true. KTrade's order is
  risk.evaluate() (hard gates) -> copilot consult only if approved -> veto -> execute.
  No change needed; verified in code.
- **#6 cycle lock:** KTrade's autonomous loop is already single-threaded and
  sequential (`time.sleep(scan_interval())` between cycles), so cycles cannot overlap
  the way Kubera's did. The relevant risk is falling behind cadence, not re-entrancy.
- **#1 advisor panel / #7 truth dashboard:** UI work. The reconciliation DATA already
  exists (the v12.8 `_broker_truth_guard` compares broker vs local positions); these
  would be a frontend surface over it. Available if you want it, but lower value than
  engine correctness and not built unprompted.

## Honest framing (unchanged)
This hardens EXECUTION discipline; it does not create an edge. The unsolved question
is still whether the daily strategy makes money — run the v13.6 fragility sweep on
your data and do the one-share Alpaca paper round-trip. Confirmation, like every gate
before it, keeps you out of obvious mistakes; it can't manufacture a profitable
strategy.
