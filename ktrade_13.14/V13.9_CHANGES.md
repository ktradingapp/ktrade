# KTrade — V13.9 (architectural execution-firewall check)

Implements the *non-theatre* version of the "external strategy guard" idea. Rather
than scanning prose for crypto buzzwords (high false positives, trivially bypassed),
it hardens the firewall KTrade already has: execution flows only through the broker
adapter, and the code must never gain a dynamic-execution path or a wallet/contract
dependency.

## Why not the proposed keyword blocklist
KTrade has NO pathway that turns external strategy *text* into executed behaviour —
no eval/exec of dynamic input, no dynamic import of uploads, no web3/wallet code; even
crypto symbols (BTC-USD/ETH-USD) trade through Alpaca. So a keyword scanner would
guard a door that doesn't exist, while false-positiving on words like "withdraw"/
"approve" and being trivially bypassed. The real protection is architectural — so the
check verifies the architecture, not the vocabulary.

## What changed: scripts/check_release_safety.py (now also a firewall assertion)
Already part of CI + the v13.8 validation pipeline. It now FAILS the build if a
*mechanism of bypass* is introduced:
- dynamic `eval()` / `exec()` of input
- `pickle` / `marshal` deserialization (RCE risk)
- `os.system(...)` / `subprocess(..., shell=True)`
- variable-driven `__import__` / `importlib.import_module`
- any wallet/contract library (web3, eth-account, ethers, solana, walletconnect, …)
  in imports OR requirements

Precision (the whole point vs a keyword scanner): it does NOT flag safe look-alikes —
`ast.literal_eval`, `df.eval(...)`, `cursor.execute(...)`, a literal
`__import__('uuid')` — nor crypto ticker strings. A legitimate flagged line can be
documented with a trailing `# release-safety: allow <reason>` comment (reported as an
explicit exception, not an error).

## New: SECURITY.md
States the policy plainly: KTrade may *analyze* external code but may only *execute*
through the approved broker adapter, and explains the architectural enforcement.

## Honest scope
This guarantees KTrade can't silently *gain* an out-of-broker execution path in code.
It is not a sandbox and not a substitute for reviewing external code before merging —
it's the automated backstop under the human review you already do. And like every
safety layer, it protects correctness, not profitability.
