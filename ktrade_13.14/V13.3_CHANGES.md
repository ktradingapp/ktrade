# KTrade — V13.3 (volatility-regime estimator — SHADOW)

The principled, testable version of the "add a Markov regime matrix" idea. It
changes NOTHING about trading; it's instrumentation plus an offline test to decide
whether the idea has any value before it ever touches sizing.

## Two deliberate choices that separate this from "add a matrix"
1. **Volatility regimes, not direction regimes.** Vol clustering (high vol begets
   high vol) is one of the most robust facts in markets; predicting return
   *direction* from a regime is fragile and overfit-prone. States are vol tertiles
   (LOW/MID/HIGH).
2. **Observable states, not latent EM.** The transition matrix is estimated from
   the classified vol states — deterministic and auditable — rather than a fitted
   HMM whose state labels flip on every refit.

## New: agent/regime_estimator.py (numpy; CLI + importable)
- `RegimeEstimator.current(prices)` → current regime, realized vol, the transition
  row P(next | current), P(escalate to HIGH), and a SUGGESTED size multiplier
  (suggestion only — never applied).
- **`evaluate_regime_value()` — the test the YouTube pitch skips.** Fits regime
  thresholds on a TRAIN window, then on HELD-OUT data compares regime-conditioned
  sizing against a FAIR constant-exposure baseline (same average exposure, so it
  isolates regime *timing* from leverage) and full exposure. Reports Sharpe and max
  drawdown for each, and says plainly whether regime sizing beats the dumb baseline
  out-of-sample. If it doesn't, "the matrix is decoration here."
- CLI: `python agent/regime_estimator.py --prices spy.csv` (CSV: date,close).

## New: shadow hook (off by default)
`KTRADE_REGIME_MODE=shadow` logs the current regime to
`logs/ktrade_regime_ledger.jsonl` each cycle (UTC), computed from the benchmark
(`KTRADE_REGIME_BENCHMARK=SPY`) in the data set. Fully guarded — never raises into
the cycle. Default `off`. Changes nothing about what trades.

## Why this is shadow-only (same discipline as the copilot)
A regime model is a hypothesis, not an edge. It tells you the likely state; it does
NOT establish that conditioning on it improves risk-adjusted return net of costs —
that must be shown out-of-sample, and regime models are notorious for looking great
in-sample and failing live. So: evaluate offline, log in shadow, and only consider
wiring it into sizing if the OOS numbers beat the baseline across multiple periods
(including a stress window like 2020). Evidence first.

Note: you already have VIX-bucket regimes, a whipsaw gate, and the copilot. This is
a *fourth* regime/risk mechanism. Adding it is easy; knowing whether any of them
helps needs the paper data you don't have yet. Don't stack more before measuring.
