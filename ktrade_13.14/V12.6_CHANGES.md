# KTrade — V12.6 Changes (production-readiness review)

Applies the V12.5 production-readiness review. All items verified against the
actual code; the three trading-path fixes are backstopped by a new regression
test (`test_v126_production_fixes.py`).

## Major (the three review items)
1. **Auto-trade never rounds sub-1-share sizing up** (`backend/ktrade_alpaca.py`).
   The autonomous path used `max(1, int(sized_qty))`, which could turn a 0.4-share
   risk approval into a 1-share order. It now floors the quantity and **skips the
   ticker** (with a logged reason) if the approved size — or the regime-multiplier-
   adjusted size — is below 1 whole share. This matches the manual `/buy` fix from
   v12.5, so both paths are consistent.
2. **Stale scan files are rejected before trading** (`backend/ktrade_alpaca.py`).
   New `_scan_payload_is_fresh()` (max age `KTRADE_MAX_SCAN_AGE_MINUTES`, default
   30). Applied in `_load_current_signals()` (auto-trade) and
   `_trusted_reference_price()` (manual BUY reference), so neither path acts on an
   outdated scan. The bundled sample scan is old, so this correctly blocks live
   trading until a fresh scan is run.
3. **`place_bracket_order()` validates defensively**. Rejects empty ticker,
   invalid side, qty < 1, non-positive stop/target, and a BUY bracket whose stop
   isn't below its target — before any broker submission. This is defense-in-depth:
   even if a caller miscomputes, the broker helper refuses a bad order.

## Fix found during validation
- **Emergency controller honors an explicit `state_file`** (`risk/emergency.py`).
  While validating, I found the kill-switch's SQLite backend ignored the
  `state_file` argument and always used the shared app-data DB — which defeated
  test isolation and made `test_v107` intermittently fail (~1 in 6) from leaked
  kill state. Now an explicit `state_file` uses file persistence; the **default
  path still uses SQLite, so production behavior is unchanged.** The full suite is
  now deterministic.
