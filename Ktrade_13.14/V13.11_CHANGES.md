# KTrade — V13.11 (wire --validate, guard --once, backtest slippage)

Three fixes from the latest external review — the ones that were verified real
against the actual code, not misreads. The review's other claims were checked and
were already handled or incorrect (see bottom).

## 1. `--validate` now runs the real validation pipeline (was a dead stub)
Previously `--validate` just printed "Validation is unavailable" and returned. It
now invokes `scripts/run_validation_pipeline.py` (the v13.8 orchestrator): compile,
release-safety, all test suites, and — with `--prices` — the backtest + fragility
stages. Verified here: `--validate` prints a READY verdict with 23 suites passing;
data stages skip cleanly without `--prices` and run on the VPS with it.

  python agent/ktrade_agent_v9.py --validate
  python agent/ktrade_agent_v9.py --validate --prices NVDA.csv --strategy macd

## 2. `--once` now runs the same pre-cycle guard as the autonomous loop
This is the accurate core of the review's VIX-staleness point. The autonomous loop
runs `_broker_truth_guard()` before every cycle (refuses new entries on a stale VIX
or a broker desync), but `--once` called `run_cycle` directly and bypassed it.
`--once` now runs the guard first. In paper mode (`KTRADE_REQUIRE_FRESH_VIX=false`)
it proceeds; with no VIX feed and the default, it stays conservative and says so.

NOTE: the review suggested putting the stale-VIX check inside `RiskEngine.evaluate()`.
That's the wrong place — it was tried earlier and broke four gate-probing test suites,
because it conflates a per-trade gate with a regime policy. Guarding `--once` (the
path that was actually unprotected) is the correct fix and touches no tests.

## 3. Backtest now models slippage, not just fees (ktrade_vectorbt.py)
`ParameterOptimizer` and `WalkForwardValidator` already applied `fees=0.001` (10bps);
they now also pass `slippage` to `vbt.Portfolio.from_signals` (default ~5bps,
`KTRADE_BACKTEST_SLIPPAGE` overrides) so backtest fills aren't optimistic. Runs on
the VPS with vectorbt+data; verified-by-construction here (compiles; can't execute
vectorbt offline).

## Verification
- `py_compile` clean. `--validate` runs the pipeline (READY, 23 suites). Full offline
  sweep: **23/23**. release-safety passes. `--validate` is read-only (takes no lock).

## On the rest of that review — checked against the real code
- "No walk-forward / Monte-Carlo / validation" — FALSE. `WalkForwardValidator` exists
  (ktrade_vectorbt.py L266), param_fragility grid sweep (v13.6), Monte-Carlo (v13.10),
  pipeline (v13.8). Only the `--validate` flag was unwired (fix #1).
- "No transaction costs" — half wrong: fees were already applied; only slippage was
  missing (fix #3).
- "Intraday scoring uses 5-DAY returns" — misread; `ret5` is 5 BARS of the scan
  interval (25 min on a 5m scan), not 5 days.
- "Copilot silently swallows errors" — it logs a warning on every error-ABSTAIN
  (L1856) and fails safe to the rules.
- "max_tokens=200 truncates reasoning" — overstated; the copilot returns a compact
  JSON verdict by design, and a truncated reply fails safe to ABSTAIN.

## Fair, but strategy choices (not built — validate on your data first)
- Time-based exit (optional, off by default): the hard stop already exits losers;
  the gap is a position chopping sideways above its stop.
- Cross-sector correlation cap: fair (AI-semis + AI-cloud are correlated), but a real
  fix needs a return-correlation matrix; a "tech complex" meta-cap is the pragmatic
  version. Medium-term.
- MACD confirmation timing: a tradeoff that directly contradicts the previous
  reviewer's push for MORE confirmation (v13.7's 2-cycle gate). Settle empirically.

## Honest framing (unchanged)
All three fixes improve correctness/plumbing/backtest-realism. None creates an edge.
The pipeline and the slippage-aware backtest are instruments to help you find out
whether the edge is real — which still requires running them on your data.
